# LightMark 面试深度准备指南

> 目标：面试时能把项目的每一部分讲清楚——不只是"做了什么"，而是"为什么这样做"、"遇到了什么问题"、"怎么解决的"。

## 一、项目一句话（电梯演讲）

LightMark 是一个基于 **Qt6 + C++17** 自研的跨平台 Markdown 桌面编辑器，支持 Windows/Linux。核心亮点：

- **自研 Markdown 解析引擎**：纯正则流水线，零第三方 Markdown 依赖，支持 10+ 种语法元素
- **虚拟化大文档编辑**：只渲染可见区域，支持万行级文档流畅编辑
- **差分渲染**：编辑时保持预览区滚动位置不跳回顶部
- **跨平台**：CMake 构建，支持 Windows/Linux

## 三、关键 Bug 修复：虚拟化编辑器滚动

这是本项目最复杂的 Bug，从发现到修好经历了三个迭代。

### 3.1 虚拟化编辑器的原理

内存中维护 `m_fullContent`（全文），QTextDocument 只存放可见区附近几十行。滚动时从 `m_fullContent` 切出新可见区写入 QTextDocument。核心矛盾：QTextEdit 内部期望拥有完整文档，但只给它一小块。

### 3.2 第一层：根本滚不动

**现象**：打开 500 行的文件，滑轮完全滚不动。`verticalScrollBar()->maximum()` 为 0。

**根因**：`setPlainText()` 只放了 ~30 行。QTextEdit 内部计算 `maximum = 30行高度 - 视口高度 ≈ 0`。滚动条 max=0，滑轮被忽略。

**修复**：在 `setFullContent()` 末尾手动 `setMaximum((全文行数-可见行数)*行高)`。

### 3.3 第二层：滚动中再次被锁

**现象**：能滚几页，然后锁死。每次 `paintEvent` 后重置。

**根因**：`paintEvent` 里也调 `setPlainText(30行)`，同样让 QTextEdit 重置滚动条为 0。

**修复**：`paintEvent` 和 `onTextChanged` 里也调 `syncScrollBarRange()`。

### 3.4 第三层：能滚动但白屏

**现象**：滚动条能滚，但下面是白屏。QTextDocument 只有 30 个块，`paintEvent` 只画 30 行。

**根因**：QTextEdit 遍历 QTextDocument 的画块（QTextBlock）。30 个块画完就是白屏。

**修复**：`rebuildFullDocument()` — 构造完整文档，文档第 i 行严格对应 `m_fullLines[i]`。可见区填真实内容，其余填空行。QTextEdit 看到完整行数的文档，滚动条自动正确，绘制也正确。

**防止数据丢失**：`MainWindow::editorFullContent()` 对 VirtualizedEditor 走 `getFullContent()`（内存全文），不走 `toPlainText()`（含空行填充）。saveFile、autoSave、updatePreview、updateWordCount 全部使用 `editorFullContent()`。

## 五、关键设计决策

### 5.1 为什么用正则流水线而非 AST？

正则流水线：Markdown 字符串按优先级逐层正则替换。优势零依赖（~400行）。AST：先拆成树结构再遍历生成 HTML（~2000行起步）。当前选择对学习项目合理。

### 5.2 为什么预览区不用 QSyntaxHighlighter？

预览区存的是 HTML（`<h1>标题</h1>`），不是 Markdown（`# 标题`）。高亮器正则匹配 Markdown 语法，拿到 HTML 匹配不到。样式用 CSS 嵌入 HTML 控制。

### 5.3 为什么不用虚拟化预览区？

预览用 `setHtml()` 全量设置，Markdown 解析极快（纯正则），瓶颈在 `setHtml()` 的 QTextDocument 重建。编辑区才是性能瓶颈（QTextDocument 维护每行布局）。

## 六、面试常见问答

**Q: 最有技术含量的部分？**
A: 虚拟化编辑器。QTextEdit 原生不支持虚拟化，我重写了 paintEvent 和滚动逻辑。核心难点是维护"屏幕视图"和"全文模型"之间的一致性——每次编辑逐行合并，每次滚动切出新区，每次查找替换在全文操作。滚动 Bug 修了三层才彻底解决。

**Q: 学到了什么？**
A: 不是"会用 API"，而是理解 API 背后的机制。`setHtml()` 会替换 QTextDocument、`QSyntaxHighlighter` 绑在 document 上、`QScrollBar::value()` 是像素不是行号、`setPlainText()` 会重置滚动条范围。

**Q: 有什么可以改进？**
A: Markdown 解析改成 AST、差分渲染做到 QTextCursor 局部替换、加插件系统支持扩展语法。

## 七、数据丢失 Bug 完整分析：为什么打开 .md 文件内容会被清空

### 7.1 现象

用 LightMark 打开一个几百行的 .md 文件，编辑区只显示一小部分内容，其余空白。滚动、编辑后，关闭再重新打开——文件里大量内容变成了空行，原本的几百行只剩几十行。

### 7.2 根因链

这个 Bug 的根因不是一行代码写错了，而是**一个设计决策的连锁反应**。整个链条分四步走：

#### 第一步：为什么要填空行？

虚拟化编辑器只把 ~30 行真实内容放入 QTextDocument。QTextEdit 内部遍历文档的 QTextBlock 来绘制——30 个块就画 30 行，画完就是白屏。滚动条虽然设对了范围，但没内容的地方就是白的。

为了解决绘制问题，采用了"空行填充"方案：把文档扩充到全文行数，可见区填真实内容，其余填空行。这样 QTextEdit 看到完整行数，绘制和滚动都对了。

#### 第二步：空行填充引入了什么副作用？

文档里这 500 行中，只有 ~30 行是真实内容，其余 ~470 行是空字符串 `""`。当任何代码调用 `QTextEdit::toPlainText()` 取编辑器内容时，它返回的是这份混了空行的文本——不是 `m_fullContent`（内存中的真正全文）。

#### 第三步：哪些操作会拿到这份脏数据？

```
saveFile()   → m_editor->toPlainText() → 空行+真实内容+空行 → 写入磁盘
autoSave()   → m_editor->toPlainText() → 同上 → 每 30 秒自动覆盖文件
updatePreview() → m_editor->toPlainText() → 同上 → 预览区显示大量空行
updateWordCount() → m_editor->toPlainText() → 同上 → 字数统计错误
```

#### 第四步：为什么第一次修复没生效？

第一次尝试在所有调用点替换为 `editorFullContent()`（走 `getFullContent()` 而非 `toPlainText()`）。但遗漏了一个关键问题：**空行填充的行号没有严格对齐**。

`getVisibleContent()` 返回的内容带了 ±2 行的 buffer。文档第 100 行本该放 `m_fullLines[100]`，实际放的是 `m_fullLines[98]`。`onTextChanged()` 也按错误偏移把用户编辑写回 `m_fullLines` 的错误位置——内容逐渐错位，最终丢失。

### 7.3 最终修复（两处）

**修复 1：精确行号对齐**

`rebuildFullDocument()` 保证**文档第 i 行严格等于 `m_fullLines[i]`**。buffer 只影响"哪些行填真实内容"，不影响行号映射：

```
文档第 0 行   = m_fullLines[0]   （可见区外→空，可见区内→真实内容）
文档第 98 行  = m_fullLines[98]  （真实内容，在 buffer 范围内）
文档第 100 行 = m_fullLines[100] （真实内容，可见区中心）
文档第 499 行 = m_fullLines[499] （可见区外→空）
```

`onTextChanged()` 用同样的精确索引读回修改。不会错位。

**修复 2：数据读取路径隔离**

新增 `MainWindow::editorFullContent()`：

```cpp
QString MainWindow::editorFullContent() const
{
    if (auto *ve = dynamic_cast<VirtualizedEditor*>(m_editor))
        return ve->getFullContent();        // 走内存全文，不碰 QTextDocument
    return m_editor->toPlainText();         // 普通 QTextEdit 走原路径
}
```

所有数据敏感路径（saveFile、autoSave、updatePreview、updateWordCount）全部切换到这个方法。QTextDocument 里的空行填充只服务于**显示层**（滚动和绘制），永远不流入**数据层**（保存和预览）。

### 7.4 教训

1. **显示层和数据层必须隔离**：QTextDocument 是显示层，`m_fullContent` 是数据层。不要用显示层的数据做持久化。
2. **行号映射必须精确**：buffer 和偏移量必须统一使用，构造和回读两边的映射公式必须一致。
3. **`setPlainText()` 会触发的副作用**：它会重置滚动条范围、发射 `textChanged`、让 Qt 内部重新计算文档布局。每次调用都要想清楚这些副作用是否会影响当前逻辑。
4. **先保证数据正确，再优化显示**：如果必须在"滚动流畅"和"内容不丢失"之间选择，永远选后者。
