## 第一章：项目是什么，为什么牛逼

### 1.1 一句话

一个用 C++ 和 Qt6 从零写的 Markdown 编辑器。**不是调包侠**——没有用现成的 Markdown 解析库，解析器是自己写的。

### 1.2 四个亮点（面试官最可能追问的）

| 亮点 | 一句话 | 面试官会问 |
|------|--------|-----------|
| **自研 Markdown 解析器** | 纯正则流水线，零依赖，~400行支持 10+ 种语法 | "为什么不用现成的库？" |
| **虚拟化编辑器** | 滚到哪渲染到哪，万行文档不卡 | "QTextEdit 不是已经支持大文档吗？" |
| **差分渲染** | 编辑时预览区不跳回顶部 | "怎么实现的？" |
| **系统性 Bug 修复** | 从 static 生命周期到滚动条像素/行号混用，15+ 个 Bug | "最有挑战性的 Bug 是哪个？" |

### 1.3 面试时这样介绍你的项目

> "我用 Qt6 和 C++17 从零写了一个 Markdown 桌面编辑器。核心亮点有三个：第一是自研的 Markdown 解析引擎，纯正则流水线，没有调任何第三方 Markdown 库；第二是虚拟化编辑器，通过重写 QTextEdit 的 paintEvent 实现只渲染可见区域，万行文档也能流畅编辑；第三是在开发过程中系统性地排查修复了 15 个以上的 Bug，对 Qt 框架的底层机制——比如 QTextDocument 生命周期、信号槽、对象树——有了比较深的理解。"

## 第三章：三条核心数据流（闭上眼能画出来）

### 3.1 流 1：用户打了一个字 → 预览更新

```
用户按下 'A'
  │
  ▼
QTextEdit 接收到按键 → 插入字符 'A' → 发射 textChanged 信号
  │
  ├──→ VirtualizedEditor::onTextChanged()
  │      toPlainText() 取可见区内容
  │      → m_visibleStartLine + i 映射 → 逐行写回 m_fullLines
  │      → m_fullContent = m_fullLines.join('\n')
  │
  ├──→ MainWindow::onTextChanged()
  │      m_isModified = true
  │      → updateWindowTitle() 加 [已修改]
  │      → 重启 m_updateTimer (200ms 防抖)
  │      → 启动 m_autoSaveDebounce (1s 后自动保存)
  │      → updateLineNumbers() 立即刷新行号
  │
  └──→ 200ms 后 m_updateTimer 超时
         → updatePreview()
              editorFullContent() → getFullContent() → 全文
              → MarkdownParser::toHtml() → HTML 片段
              → 包裹亮色/暗色 CSS → 完整 HTML
              → m_preview->setHtml()
              → 保持或重置滚动条位置
         → updateWordCount()  中文按字/英文按词统计
         → updateStatusBar()  光标行列 + 文件信息
```

### 3.2 流 2：打开一个 500 行的 .md 文件

```
MainWindow::openFile()
  → QFileDialog 选文件
  → loadFile(fileName)
       → QFile 读文件 → data (QByteArray)
       → FileEncodingDetector 检测编码 → encoding (UTF-8/GBK/UTF-16LE)
       → 按编码解码 → content (QString)
       → vEditor->setFullContent(content)
            m_fullContent = content
            m_fullLines = content.split('\n')
            updateVisibleLines() → 初始可见区 0~30 行
            rebuildFullDocument() → 空行填充 + 可见内容 + 空行填充
            setPlainText() → QTextEdit 看到 500 行 → 滚动条正确
       → setCurrentFile(fileName) → 更新标题栏
       → m_isFirstPreviewRender = true → 下次预览从头开始
```

### 3.3 流 3：用户滚轮滚动

```
用户滚轮
  → QScrollBar::valueChanged 信号
  → updateVisibleLines()
       scrollValue / lineHeight → m_visibleStartLine (像素→行号)
  → paintEvent()
       发现 m_visibleStartLine != m_lastVisibleStart
       → 保存当前滚动位置
       → rebuildFullDocument()  // 用新的可见区重建文档
           文档[0..start-1] = 空行
           文档[start..end] = m_fullLines[start..end] (真实内容)
           文档[end+1..total-1] = 空行
           setPlainText() → QTextEdit 更新
       → 恢复光标和滚动位置
  → 用户看到新区域的内容
```

## 第四章：每个模块深度讲解

### 4.1 MarkdownParser —— 你的解析引擎

**入口函数**：`MarkdownParser::toHtml(markdownText)`

**解析顺序（为什么这个顺序很重要）**：
```
1. parseBlockCode()     ← 最先：代码块里可能有 # ** 等符号，不能被后面的规则误匹配
2. parseTables()        ← 表格也是块级，里面有 | 等特殊字符
3. parseHeaders()       ← # 转 <h1>~<h6>
4. parseLists()         ← - 和 1. 转 <ul>/<ol>
5. parseBoldAndItalic() ← **粗体** *斜体*（行内元素）
6. parseInlineCode()    ← `代码`（行内）
7. parseLinks()         ← [文字](url)
8. parseImages()        ← ![alt](url)
9. parseLineBreaks()    ← 最后：\n\n → </p><p>，\n → <br>
```

**为什么代码块最先？** 代码块 `` ``` `` 里面可能写 `# 这不是标题 **这也不是粗体**`。如果先处理标题和粗体，代码块里的内容会被错误转换。

**为什么换行最后？** 前面已经生成了 `<h1>`、`<pre>`、`<ul>` 等 HTML 标签。换行处理需要识别这些已有的标签，避免在标签内部插入 `<br>`。所以它用"占位符保护法"——先把已有 HTML 标签替换成 `__BLOCK_0__` 占位符 → 对剩余文本处理换行 → 再把占位符换回来。

### 4.2 VirtualizedEditor —— 你的虚拟化引擎

**为什么需要它？** QTextEdit 在加载几千行文档时会卡。因为 QTextDocument 对每一行都维护布局信息（行高、换行位置等）。行数越多，布局计算越重。

**你怎么解决的？** 全文存在 `m_fullContent`（一个 QString）里。QTextDocument 里永远只放可见区附近几十行。滚动时从 `m_fullContent` 切出新内容放进去。编辑时把修改逐行合并回 `m_fullContent`。

**核心方法**：

| 方法 | 做什么 | 何时调用 |
|------|--------|---------|
| `setFullContent()` | 把全文加载到内存 | 打开文件时 |
| `updateVisibleLines()` | 根据滚动条像素值算当前在哪几行 | 每次滚动 |
| `rebuildFullDocument()` | 构造完整文档（空行+内容+空行）→setPlainText | 滚动到新区域时 |
| `onTextChanged()` | 把 QTextDocument 里的编辑合并回 m_fullContent | 每次打字 |

**关键数据结构**：
```
m_fullContent = "第1行\n第2行\n...\n第1000行"  ← 全文，一个 QString
m_fullLines   = ["第1行", "第2行", ..., "第1000行"]  ← 按 \n 拆分，用来按行号索引

m_visibleStartLine = 100   ← 当前屏幕顶部是全文第几行
m_visibleEndLine   = 130   ← 当前屏幕底部是全文第几行
m_linesPerView     = 30    ← 一屏能显示多少行
```

### 4.3 编辑区高亮器 —— EditorHighlighter

继承 `QSyntaxHighlighter`，重写一个函数：`highlightBlock(const QString &text)`。

Qt 对文档的每一行调用一次这个函数。你的工作是：检查这一行文本匹不匹配 Markdown 语法，匹配就给对应字符范围调 `setFormat()` 染色。

```
输入 text = "## 这是**标题**"
  → 匹配到 "## " → setFormat(0, 3, headingFormat) → 深蓝色加粗
  → 匹配到 "**标题**" → setFormat(4, 6, boldFormat) → 深绿色加粗
```

**Bug 4 的故事**：原来 `inCodeBlock` 是 `static bool`，导致所有文档共享同一个代码块状态。先打开含代码块的 A，再新建 B——B 的第一行就被当成代码块染色。改成成员变量后每个文档独立。

### 4.4 预览区样式 —— CSS

预览区不是用 QSyntaxHighlighter 染色的（因为里面是 HTML 不是 Markdown）。而是在 `updatePreview()` 里把一份完整的 CSS 嵌入 HTML 的 `<style>` 标签里。

CSS 覆盖了 8 种元素：body、h1~h3、code、pre、a、ul/ol、blockquote、img、table。亮色和暗色各一套，通过 `m_darkThemeEnabled` 切换。

## 第五章：Bug 故事集（面试必讲）

### 5.1 最值的讲的：虚拟化编辑器滚动三迭代

**第一层**：打开 500 行文件滚不动 → 发现 `setPlainText(30行)` → QTextEdit 内部 `maximum = 30行高度 - 视口 ≈ 0` → `setMaximum()` 手动修正。

**第二层**：能滚几页又锁死 → 发现 `paintEvent()` 里也调 `setPlainText(30行)` → 同样重置滚动条 → 每次滚动都调 `syncScrollBarRange()`。

**第三层**：能滚但白屏 → 发现 QTextEdit 只画 QTextDocument 里实际存在的块（30 个），30 块画完就是白的 → `rebuildFullDocument()` 空行填充到全文行数 → QTextEdit 看到 500 块 → 滚动条和绘制都对了。

### 5.2 讲了显示你懂 C++ 的：static 生命周期

代码块高亮状态 `inCodeBlock` 是 `static bool`——整个程序只有一份。打开文件 A 设成 true，新建文件 B 还是 true→文件 B 从第一行就被当成代码块染色。改成成员变量，每个高亮器实例独立维护。

### 5.3 讲了显示你懂 Qt 的：预览高亮器失效

`QSyntaxHighlighter` 绑在 `QTextDocument` 上。`setHtml()` 会替换整个 document→旧 document 销毁→高亮器跟着销毁。即使没销毁，高亮器正则是匹配 Markdown 的，预览区是 HTML，根本匹配不到。

### 5.4 讲了显示你细心：菜单索引

用 `fileActions[0]`、`fileActions[1]` 连接信号槽。如果在菜单中间插入一项，所有索引偏移→功能连接到错误槽函数→编译器不报错。全部改成成员变量按名字连接。

### 5.5 数据丢失（最严重的 Bug）

空行填充方案让 `toPlainText()` 返回混了空行的内容。saveFile、autoSave 直接把空行写入磁盘→原始内容被覆盖。修复：所有数据读取路径走 `editorFullContent()`（取 `getFullContent()` 内存全文），QTextDocument 只用于显示。

## 第八章：VirtualizedEditor 逐行讲解

> 这一章是最硬核的。读完之后你不仅能理解这个文件，还能自己写一个。

整个文件 161 行，分 7 个函数。先看头文件里定义了哪些数据成员，再看每个函数怎么操作它们。

### 8.0 头文件：数据成员的含义

```cpp
class VirtualizedEditor : public QTextEdit    // 继承 Qt 的标准文本编辑器
{
    QString m_fullContent;      // 全文文本，一个长字符串 "行1\n行2\n...\n行1000"
    QStringList m_fullLines;    // 全文按 \n 拆开的行数组 ["行1","行2",...,"行1000"]
                                //   注意：m_fullContent 和 m_fullLines 存的是同一份数据
                                //   m_fullLines 只是为了能按行号 O(1) 访问某一行
                                //   编辑后两者要同步更新

    int m_visibleStartLine;     // 当前屏幕顶部是全文的第几行（0-based）
    int m_visibleEndLine;       // 当前屏幕底部是全文的第几行（0-based）
    int m_linesPerView;         // 当前窗口高度能显示多少行
                                //   计算公式: height() / 行高，在 paintEvent 中更新

    int m_globalCursorLine;     // 光标在全文中的行号（不是屏幕上的行号！）
    int m_globalCursorColumn;   // 光标在当前行的第几列

    int m_lastVisibleStart = -1; // 上一次 paintEvent 时的可见起始行
                                  //   -1 确保第一次 paintEvent 一定会触发重建

    bool m_isProgrammaticChange = false; // true = 这次文档变化是程序主动触发的
                                         //   （如滚动切换可见区），不是用户打字
                                         //   用来在 onTextChanged 中区分"程序改的"和"用户改的"
};
```

**核心数据关系图**：
```
全文（内存）:
  m_fullContent = "第1行\n第2行\n...\n第1000行"
  m_fullLines   = ["第1行", "第2行", ..., "第1000行"]

屏幕（QTextDocument）:
  只存放 ~30 行，位置由 m_visibleStartLine 决定
  m_visibleStartLine = 100 → 屏幕显示第 100~129 行

  用户看到的 = m_fullLines[m_visibleStartLine .. m_visibleEndLine]
  用户编辑的 = QTextDocument 里的内容 → onTextChanged 合并回去
```

### 8.1 构造函数（第 6-21 行）

```cpp

**接下来的三个 connect 是核心事件链**：

```cpp

**这是第 1 个 connect**：滚动条的值变了 → 调 `updateVisibleLines()`。

```cpp

            &VirtualizedEditor::onTextChanged);
```

**这是第 2 个 connect**：文档内容变了（用户打字）→ 调 `onTextChanged()`。

QTextEdit 的 `textChanged` 信号在**任何**文本变化时发射，包括用户打字、粘贴，也包括我们程序里调 `setPlainText()`。所以 `onTextChanged` 里第一行就是检查 `m_isProgrammaticChange` 标志来区分"程序改的"和"用户改的"。

```cpp

        QTextCursor cursor = textCursor();
        m_globalCursorLine = m_visibleStartLine + cursor.blockNumber();
        m_globalCursorColumn = cursor.columnNumber();

}
```

**这是第 3 个 connect**：光标位置变了 → 更新全局光标坐标。

为什么需要这个？因为 QTextDocument 里只有可见区 ~30 行，光标在 QTextDocument 里的位置是"相对屏幕左上角的行号"（范围 0~30）。但我们需要的是"光标在全文里的行号"（范围 0~1000）。

转换公式：`全局行号 = 屏幕顶部行号 + 光标在屏幕上的行号`。

例如：屏幕顶部是全文第 100 行，光标在屏幕上第 5 行 → 全局行号 = 100 + 5 = 105。

`cursor.blockNumber()` 返回光标在当前 QTextDocument 里的块编号（0-based）。注意"块编号"就是行号——QTextDocument 里每个 QTextBlock 对应一行。

### 8.2 setFullContent（第 23-37 行）

**作用**：打开文件时调用，把文件的全部文本加载到内存，然后让屏幕上显示第一页。

**为什么同时存 String 和 StringList？** 因为不同操作需要不同形式：

| 操作 | 用哪个 | 原因 |
|------|--------|------|
| 按行号取第 100 行 | m_fullLines[100] | O(1) 直接索引 |
| 全文搜索替换 | m_fullContent | indexOf/replace 操作方便 |

| 传给 MarkdownParser | m_fullContent | 它需要完整字符串 |

`QFontMetrics::lineSpacing()` 返回"一行文字占多少像素"，包括行间距。`height()` 是控件总高度。`height() / lineSpacing()` = 一屏能显示多少行。

### 8.5 updateVisibleLines（第 81-101 行）

**作用**：把滚动条的**像素位置**转换为"全文中的行号"。这个函数只计算、不改 UI。

```cpp
void VirtualizedEditor::updateVisibleLines()
{
    if (m_fullLines.isEmpty()) {
        m_visibleStartLine = 0;
        m_visibleEndLine = 0;
        return;
    }
```

**边界防御**：空文档特殊处理。全文行数为 0 时，后面 `m_fullLines.size()` 是 0，所有计算都没意义。

```cpp
    QFontMetrics fm(font());
    int lineHeight = fm.lineSpacing();
    if (lineHeight <= 0) lineHeight = 16;
```

**获取行高**。`lineSpacing()` 返回一行文字占的像素高度。如果字体信息还没加载（罕见但可能），fallback 到 16px——这是一个常见等宽字体的默认行高。

```cpp
    int scrollValue = verticalScrollBar()->value();
    m_visibleStartLine = scrollValue / lineHeight;
```

**像素→行号转换**。这就是之前 Bug 2 的根因——原来写的是 `m_visibleStartLine = scrollValue`（直接把像素当行号），少除了 `lineHeight`。

原理：滚动条 value 表示"文档顶部被推上去多少像素"。如果 value = 1600px，行高 = 16px → 1600/16 = 100 → 屏幕顶部现在是全文第 100 行。

```cpp
    m_visibleEndLine = m_visibleStartLine + m_linesPerView - 1;
```

**屏幕底部行号 = 顶部行号 + 一屏能显示的行数 - 1**。减 1 是因为行号是 0-based 的——如果 m_linesPerView=30，顶部是第 100 行，底部是第 129 行（共 30 行）。

```cpp
    if (m_visibleEndLine >= m_fullLines.size())
        m_visibleEndLine = m_fullLines.size() - 1;
    if (m_visibleStartLine >= m_fullLines.size())
        m_visibleStartLine = qMax(0, m_fullLines.size() - m_linesPerView);
}
```

**边界钳制**：用户滚到文档末尾时，可见区的起止行可能超出全文范围。钳制到有效范围内。`qMax(0, ...)` 防止起始行变负数。

### 8.6 rebuildFullDocument（第 103-129 行）

**作用**：构造一份"块数 = 全文行数"的完整文档，可见区放真实内容，其余填空行。这是解决"能滚动但白屏"的关键。

```cpp
void VirtualizedEditor::rebuildFullDocument()
{
    const int total = m_fullLines.size();
```

全文总行数。假设 500。

```cpp
    const int buf = 2;
```

**buffer 行数**。可见区上下各多渲染 2 行，这样用户缓慢滚动 1-2 行时不会触发重建——已经有内容在文档里了，减少了 `setPlainText` 的调用频率。

```cpp
    int contentStart = qMax(0, m_visibleStartLine - buf);
    int contentEnd = qMin(total - 1, m_visibleEndLine + buf);
```

**计算哪几行放真实内容**。假设 `m_visibleStartLine = 100, m_visibleEndLine = 129`：
- contentStart = max(0, 100-2) = 98
- contentEnd = min(499, 129+2) = 131

所以真实内容覆盖第 98~131 行（共 34 行），比可见区上下各多了 2 行 buffer。

```cpp
    QStringList docLines;
    docLines.reserve(total);
```

**预分配内存**。`reserve(total)` 让 QList 一次性申请 500 个元素的空间，避免循环中反复扩容。对于 10000 行文档，不 reserve 的话可能要扩容十几次。

```cpp
    for (int i = 0; i < contentStart; ++i)
        docLines << QString();
```

**顶部空行填充**：行 0~97 填空字符串。`QString()` 构造空串。

```cpp
    for (int i = contentStart; i <= contentEnd; ++i)
        docLines << m_fullLines[i];
```

**真实内容**：行 98~131 放 `m_fullLines[98]` 到 `m_fullLines[131]`。**文档行号和全文行号严格对应**——文档第 98 行 = m_fullLines[98]。这就是"精确对齐"。

```cpp
    for (int i = contentEnd + 1; i < total; ++i)
        docLines << QString();
```

**底部空行填充**：行 132~499 填空串。

```cpp
    setPlainText(docLines.join('\n'));
}
```

**一次性设入**。`join('\n')` 把 500 行拼成一个带换行符的长字符串。`setPlainText()` 把这个字符串交给 QTextEdit→QTextDocument 解析成 500 个 QTextBlock。

QTextEdit 看到 500 行内容 → 自动把滚动条最大值设为 (500行高度 - 视口高度) → 滚动条完全正确。`paintEvent` 遍历 500 个 block，空行画空白，真实行画文字 → 绘制完全正确。

**性能**：QString 拼接 500 行（其中 466 行是空串）在微秒级。QTextDocument 建立 500 个 block 的布局在毫秒级。对于 10000 行文档也完全可接受。

### 8.7 onTextChanged（第 131-160 行）

**作用**：用户打了一个字后，Qt 发射 `textChanged` 信号到这里。我们把 QTextDocument 里的改动合并回 `m_fullContent`。

```cpp
void VirtualizedEditor::onTextChanged()
{
    if (m_isProgrammaticChange) return;
```

**程序改的不管**。如果是 `rebuildFullDocument()` 或 `setFullContent()` 触发的 textChanged（因为它们都设了 `m_isProgrammaticChange = true`），直接返回。这些情况不需要合并——内容就是从 m_fullContent 来的，再合并回去等于原地打转。

```cpp
    QStringList allDocLines = toPlainText().split('\n');
```

从 QTextDocument 读取完整文档内容（包含空行填充），按行拆分。

**注意这里的关键**：`toPlainText()` 返回的是含空行填充的完整文档（500 行），不是只可见区。这是因为 QTextDocument 里存的就是空行+内容+空行的完整文档。

```cpp
    const int buf = 2;
    int contentStart = qMax(0, m_visibleStartLine - buf);
    int contentEnd = qMin(m_fullLines.size() - 1, m_visibleEndLine + buf);
```

计算"哪几行是真实内容"——和 `rebuildFullDocument()` 用完全一样的公式。**这是之前数据丢失 Bug 的修复点——这里的映射必须和重建时的映射一致**。

```cpp
    QStringList newVisible;
    for (int i = contentStart; i <= contentEnd && i < allDocLines.size(); ++i)
        newVisible << allDocLines[i];
    QStringList oldVisible;
    for (int i = contentStart; i <= contentEnd && i < m_fullLines.size(); ++i)
        oldVisible << m_fullLines[i];
    if (newVisible.join('\n') == oldVisible.join('\n')) return;
```

**先比较再决定要不要合并**。从 QTextDocument 中提取可见区的真实行（newVisible），和 m_fullLines 中对应的行（oldVisible）比较。如果一样——没有真正的编辑发生（可能是 Qt 内部触发的 textChanged，比如高亮器重新染色）——直接返回，避免无意义的合并。

`i < allDocLines.size()` 和 `i < m_fullLines.size()` 是防御：理论上文档行数 = 全文行数，但万一不一致，不越界。

```cpp
    QStringList fullLines = m_fullContent.split('\n');
    for (int i = contentStart; i <= contentEnd && i < allDocLines.size(); ++i) {
        if (i < fullLines.size())
            fullLines[i] = allDocLines[i];
        else
            fullLines.append(allDocLines[i]);
    }
    m_fullContent = fullLines.join('\n');
    m_fullLines = fullLines;
}
```

**逐行写回**。行号从 contentStart 到 contentEnd——**只看可见区范围**。不在可见区内的行保持不变（没有被用户修改，也不需要更新）。

`if (i < fullLines.size())` → 行已存在，替换内容。
`else fullLines.append(...)` → 不存在，追加（用户在文档末尾回车新建了一行）。

最后同时更新 `m_fullContent`（字符串形式）和 `m_fullLines`（数组形式），保持两者一致。

### 8.8 如果你要自己写一个虚拟化编辑器

记住这六个要点：

1. **数据分两层**：内存全文（m_fullContent）+ 屏幕视图（QTextDocument）
2. **滚动 = 重新切内容**：scrollBar value 变了 → 算可见行号 → 重选内容 → setPlainText
3. **编辑 = 合并回去**：textChanged → 从 QTextDocument 读 → 写回 m_fullContent
4. **程序改的和用户改的要区分**：flag 防止循环
5. **光标要全局化**：屏幕行号 + m_visibleStartLine = 全文行号
6. **文档块数和全文行数要一致**：空行填充让 QTextEdit 的滚动条和绘制都正确

## 第九章：VirtualizedEditor 关键 Bug 修复记录

### 9.1 Bug：新建空白文档时预览区不渲染

**现象**：启动程序后直接打字（不打开已有文件），编辑区正常显示但右侧预览区始终空白。

**根因**：

`VirtualizedEditor::onTextChanged()` 通过比较"可见区新旧内容"来合并用户编辑。它用 `m_fullLines.size()` 计算可见行范围：

```cpp
int contentEnd = qMin(m_fullLines.size() - 1, m_visibleEndLine + buf);
```

新建文档时 `m_fullLines` 为空（`size() == 0`），导致 `contentEnd = -1`。合并循环 `for (i = 0; i <= -1; i++)` 一次都不执行，`newVisible` 和 `oldVisible` 都是空串，相等，直接 return。**`m_fullContent` 永远得不到用户的输入内容**，预览区拿到的始终是空字符串。

修复前代码路径：
```
用户输入 "# Hello"
  → onTextChanged()
  → m_fullLines.size() = 0
  → contentEnd = min(-1, ...) = -1
  → 合并循环体不执行
  → newVisible == oldVisible（都是空）→ Early Return
  → m_fullContent 仍为 ""
  → updatePreview() → editorFullContent() → "" → 预览空白
```

**修复**：改用 QTextEdit 当前文档全文（`toPlainText().split('\n')`）作为 `allDocLines` 的来源，而非依赖旧 `m_fullLines` 的行数范围。核心改动：

```cpp
// 修改前：contentEnd 受限于旧的 m_fullLines 大小
int contentEnd = qMin(m_fullLines.size() - 1, m_visibleEndLine + buf);

// 修改后：contentEnd 基于当前文档实际行数
int contentEnd = qMin(m_visibleEndLine + bufferLines, allDocLines.size() - 1);
```

同时增加 `contentEnd < contentStart` 的判空保护——当全选删除后文档只剩一个空行时，直接清空全文模型。

### 9.2 Bug：全选多行内容后删除，预览区只清掉第一行

**现象**：编辑器有 5 行内容，Ctrl+A 全选、按 Delete 删除。预览区只把第 1 行删掉了，剩下 4 行还在。

**根因**：合并循环 `for (i = contentStart; i <= contentEnd; i++)` 只更新了可见缓冲区范围内的行。全选删除后文档只剩 1 个空行（`allDocLines = [""]`），`contentEnd = 0`，循环只更新了 `fullLines[0]`。但 `fullLines` 从 `m_fullLines` 拷贝而来有 5 行，下标 1-4 的数据没有被清理。

修复前代码路径：
```
全选 5 行 → Delete
  → toPlainText() = "" → allDocLines = [""]（1行）
  → contentStart = 0, contentEnd = 0
  → 合并循环：fullLines[0] = "" ✓
  → fullLines[1..4] 残留旧数据 ✗
  → m_fullContent = fullLines.join('\n') → 5 行（第1行空，其余4行旧内容）
  → 预览区显示旧内容
```

**修复**：合并循环后增加行数对齐逻辑——文档变小时 `removeLast()` 裁掉多余行，文档变大时补空行。

```cpp
// 合并循环后
while (fullLines.size() < allDocLines.size()) {
    fullLines.append(QString());      // 文档变大：补齐空行
}
while (fullLines.size() > allDocLines.size()) {
    fullLines.removeLast();           // 文档变小：裁掉多余
}
```

### 9.3 面试怎么讲这个 Bug

> 我做虚拟化编辑器的时候遇到过一个比较隐蔽的 bug：新建空白文档直接打字，预览区不渲染。排查发现 `onTextChanged` 里用旧模型的行数去算合并范围，新文档旧模型是空的，`contentEnd` 算出来是 -1，合并逻辑根本不执行。改了用当前文档的实际行数做范围上限之后就好了。后来还发现全选删除的类似问题——合并循环只更新可见区内的行，文档缩小后多余行残留了，加了行数对齐裁掉多余行就解决了。
