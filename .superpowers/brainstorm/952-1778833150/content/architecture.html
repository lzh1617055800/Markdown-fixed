这 4 个模块各干什么

1. ChatPanel（UI 层）

QWidget，嵌在 QSplitter 底部。职责只有一件事：显示对话和捕获输入。

  • 内部用 QScrollArea + 多个 ChatBubble（自定义 QWidget）展示消息
  • 每个气泡有"角色"（用户/AI）、"状态"（等待中/流式中/完成）
  • 输入框 + 发送按钮，Ctrl+Enter 快捷发送
  • 收到 AiClient 的流式信号时，追加重绘当前气泡
  • 不知道任何 HTTP/API 细节

2. ChatModel（数据层）

存消息列表和对话状态，纯数据类，不依赖 Qt Widget。

  • 消息结构体：角色（User/Assistant/System）、内容、时间戳、状态
  • 对话历史 List，用于发给 AI 做上下文
  • 支持清空、导出（Markdown 格式）
  • 和 UI 完全解耦

3. AiClient（抽象接口）

定义统一的 AI 调用接口，Mock 和真实实现都继承它。

  • 纯虚基类，定义 sendMessage(QString userText)
  • 信号：streamChunk(QString chunk) / responseDone() / errorOccurred(QString)
  • MockAiClient：用 QTimer 模拟打字机效果，一行一行吐出预置文本
  • StreamAiClient（后期）：用 QNetworkAccessManager 发 HTTPS POST 到 API，逐行解析 SSE
  • 替换后端时只改这一个地方

4. ContextBuilder

把编辑器的当前内容 + 用户问题拼成 AI 能理解的消息。

  • System 角色：固定提示词（"你是一个 Markdown 写作助手..."）
  • User 角色：当前文档全文 + 用户的问题
  • 剪裁策略：文档太长时截断，优先保留开头和光标附近
  • 预置快捷操作："帮我优化这段文字"、"检查语法错误"、"翻译成英文"

数据流：用户发一条消息的完整路径

核心挑战：流式打字机效果怎么实现

这是整个 AI 面板体验最关键的技术点。用户应该看到 AI 的回复一个字一个字出来，而不是等 5 秒后突然蹦出一大段。

Mock 阶段：QTimer 模拟

  • 预设一段完整回复文本，存为 QStringList（按句拆分）
  • QTimer 每 200-300ms 触发一次，取下一句，emit streamChunk
  • ChatPanel 收到 chunk 后 append 到当前 AI 气泡的 QLabel
  • 定时器耗尽所有句子后 emit responseDone

真实 API 阶段：SSE 流解析

  • Anthropic API 返回 text/event-stream
  • QNetworkReply::readyRead 信号每次有数据到达时触发
  • 增量解析 SSE 行：data: {"type":"content_block_delta","delta":{"text":"一段"}}
  • 提取 delta.text，emit streamChunk
  • 关键：不阻塞 UI 线程——网络回调在 Qt 事件循环里异步执行
